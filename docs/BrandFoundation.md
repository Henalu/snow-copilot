# SN Assistant Brand Foundation

Last updated: 2026-03-24

## Why SN Assistant exists

ServiceNow developers lose time in two ways:

- understanding platform-specific code and changes that were not written by them
- fighting generic AI answers that sound polished but are not actually valid for ServiceNow

SN Assistant exists to close that gap.

It is built to feel like a practical copilot inside the platform, not a detached chatbot outside of it.

## Audience

Primary users:

- ServiceNow developers
- technical consultants
- solution engineers
- architects doing implementation review

Secondary users:

- technical leads reviewing inherited code
- teams preparing handover or delivery documentation
- consultants documenting Update Sets for clients or internal stakeholders

## Jobs to be done

Users come to SN Assistant to:

- understand what a script really does
- avoid invalid ServiceNow patterns
- document work faster without losing technical quality
- explain and justify a change package
- move quickly without trusting generic AI blindly

## Positioning

SN Assistant is a ServiceNow-native AI copilot for explaining, improving, and documenting platform code and changes.

Unlike generic AI browser tools, it is designed around ServiceNow constraints, ServiceNow artifacts, and ServiceNow implementation reality.

## Vision

A world where every ServiceNow team can reason about code and change with confidence, speed, and technical honesty.

## Mission

Help ServiceNow developers move faster with grounded, useful assistance that respects the platform, the developer's judgment, and the cost of getting technical details wrong.

## Product promise

SN Assistant should help users:

- get to a useful answer quickly
- stay closer to valid ServiceNow practice
- see uncertainty when certainty would be misleading
- produce outputs that are immediately usable in real delivery work

## Values

### 1. Platform truth over generic cleverness

If an answer sounds good but is invalid in ServiceNow, it is not good enough.

### 2. Speed is part of quality

A copilot that is technically right but painfully slow still fails the real job. Responsiveness matters.

### 3. Honesty over confident guessing

The product should surface uncertainty when evidence is weak, instead of inventing authority.

### 4. Utility over performance theater

The goal is not to impress with fancy output. The goal is to help someone ship, review, explain, or document real work.

### 5. Respect the developer's control

SN Assistant assists. It does not pretend to replace judgment, review, or platform expertise.

### 6. Calm expertise

The product should feel like a strong technical teammate: clear, capable, direct, and never patronizing.

## Personality

If SN Assistant were a person, it would be:

- technically sharp
- pragmatic
- calm under pressure
- specific instead of vague
- helpful without ego
- honest about limits

## Tone of voice

### We should sound like this

- clear
- direct
- technically grounded
- warm but not fluffy
- concise by default
- transparent when something is uncertain

### We should avoid sounding like this

- generic AI hype
- corporate filler
- overconfident without evidence
- theatrical or salesy
- too academic for practical work

## What we are

- a ServiceNow-specific copilot
- a productivity tool
- a documentation and reasoning assistant
- a platform-aware layer on top of user-selected AI providers

## What we are not

- a guaranteed source of truth
- a replacement for testing or code review
- a generic "ask any model" shell with no platform opinion
- an autopilot that should be trusted blindly

## Trust principles

To feel credible in public, SN Assistant should always make these things legible:

- what contexts are truly supported
- what parts are experimental
- where data goes
- when external AI providers are involved
- when the answer is grounded versus inferred
